实验过程

RTFSC

在~/ics2021/nemu/include/cpu中

  • Operand

    • IFDEF

      #ifdef预处理程序指令检查宏是否由#define定义。 如果是，则执行代码，否则#else代码执行(如果存在)。

      在定义结构体时进行宏定义，其作用域从定义处开始直至文件结束。

    •  rtlreg_t *preg; //直接指向那些已经存在的RTL寄存器
    word_t imm; //无符号立即数
    sword_t simm; //有符号立即数

在nemu/src/isa/riscv-32/include/isa-def.h中

  • ISADecodeInfo

    对ISA相关的信息进行抽象。

    Struct内有union instr，instr内部的struct有i, s, u, 除了这些struct还有一个val数。

    用uint32_t数的形式对信息进行抽象。

在nemu/src/cpu/cpu-exe.c中

  • fetch_decode()

    先把当前的PC保存到s的成员pc和snpc中, 其中s->pc就是当前指令的PC, 而s->snpc则是下一条指令的PC, 这里的snpc是"static next PC"的意思.

在decode.c中

  • isa_fetch_decode

    会随着取指的过程修改s->snpc的值, 使得从isa_fetch_decode()返回后s->snpc正好为下一条指令的PC.

    返回一个编号idx, 用于对g_exec_table这一数组进行索引

    g_exec_table是一个函数指针的数组, 数组中的每个元素都会指向一个用于模拟指令执行的函数, 我们把这样的函数称为"执行辅助函数"(execution helper function). 通过idx索引这个数组, 可以找到与一条指令相匹配的执行辅助函数, 并把它记录到s->EHelper中

在cpu-exe.c中

  • fetch_decode_exec_updatepc()

    接受一个Decode类型的结构体指针s

    然后代码会调用isa_fetch_decode()函数

    会调用刚才记录到s->EHelper的执行辅助函数, 来模拟指令执行的真正操作. 最后会更新PC, 让PC指向下一条指令.

在ifetch.h中

  • instr_fetch(vaddr_t pc, int len)

    负责取指令的工作.

    会根据参数len来调用vaddr_ifetch()

    • vaddr_ifetch()

      通过paddr_read()来访问物理内存中的内容

    • 取指操作的本质只不过就是一次内存的访问而已

…sc 实在太多，其中比较重要的还是前面两个结构，在实现指令的时候会作为操作对象。

关于伪指令

以为伪指令也要实现，所以傻傻地把什么li和j的过程也写了，还为匹配规则怎么写，感到迷惑，后来查了伪指令的定义才恍然大悟：该类指令并不是可执行指令，没有机器代码，只用于汇编过程中为汇编程序提供汇编信息，原来是指令，又不完全是指令。

为什么会浪费大量的时间找不到方向，还总是做一些无用功呢？当然是没有好好rtfm和rtfsc啦。虽然上个学期学了点RISC-V32，还做了实验，但是也忘得差不多了，所以把手册又翻出来读了，不过比较功利地选择了中文手册，不过里面的附录是真的有用。

和x86比起来，riscv确实要精简不少。把指令全部实现也许也不是什么难事（好大的口气

运行dummy.c

根据txt文件里的反汇编结果，需要实现的指令应该是auipc, addi和jal，lui指令(compute.h)和sw指令(ldst.h)在框架中已经被实现。

根据讲义，要想添加新的指令：

  • 在nemu/src/isa/riscv32/instr/decode.c中添加正确的模式匹配规则。

    将要用的指令添加在了表格辅助函数def_THelper(main)下，根据txt内的反汇编结果确定了七位opcode。

  • 用RTL实现正确的执行辅助函数, 需要注意使用RTL伪指令时要遵守上文提到的小型调用约定。

    • auipc(compute.h)：向pc高位加上立即数

      查阅手册得到应该实现的功能是：

      x[rd] = pc + sext(immediate[31:12] << 12)

      要注意的是，addi的操作数必须要一个是立即数一个是寄存器，所以可以用x0作为周转。

    • addi(compute.h)

      查阅手册得到应该实现的功能是：

      x[rd] = x[rs1] + sext(immediate)

    • jal(control.h)

      查阅手册得到实现为：

      x[rd] = pc + 4;

      pc += sext(offset);

      其中需要进行符号扩展，在rtl伪指令(pseudo.h)中有rtl_sext，但是需要自己实现。(在真正实现jal的时候)

  • 在nemu/src/isa/riscv32/include/isa-all-instr.h中把指令添加到INSTR_LIST中

  • 必要时在nemu/src/isa/riscv32/include/isa-exec.h中添加相应的头文件

做完这些后就将dummy在nemu中make run，出现了大量error…

第一个问题是在auipc的匹配规则里少了一个逗号，属于脑瘫问题。

第二个问题看了好几集，报错信息显示在decode.h里，但按理来说我也没修改decode.h，所以在decode里找和decode_J有关的错误，然后发现是J型的译码辅助函数没有写。根据手册，实现译码辅助函数。通过手册我们可以知道，J型指令是U型指令的一个“变种”，所以可以考虑通过U型指令来获取想要的信息。

再次make run，报错，因为<<运算和+运算的优先级，缺少括号。

再次make run FAIL，再改，再make，再改…我爬了。

显示是有指令没有执行，我想了想估计还是伪指令的问题，就又回去读了手册。

对于txt文件中的伪指令：

  • li：等同于执行lui和/或addi
  • mv：实际被扩展为addi rd，rs1，0
  • ret：实际被扩展为jalr x0，0(x1)
  • j：等同于jal x0，offset

检查了一下，还缺一个jalr指令

jalr(control.h)

实际上是jal的寄存器版本。把pc设置为x[rs1]+sext(offset)，把计算出的地址的最低有效位设为0，并将原pc+4的值写入f[rd]。rd默认为x1。

根据手册需要实现t = pc + 4; pc = (x[rs1] + sext(offset)) & ~1; x[rd] = t;

但是…就是说，还是FAIL了，然后开始根据讲义仔细阅读abort at的pc，看了txt好像没有这个指令。

啊，原来是jal写错了啊，TT。

那些试验中我跳错过的位置：

  • 0x8000180c、0x80000c0c：J型译码辅助函数的立即数读取问题，在计算最终offset值时移位有错，已解决。
  • 0x80000012：仍然是译码函数立即数读取的问题，已解决。

最后dummy终于成功运行了。

但是按照之前犯过的一些错误，jarl指令此时应该也是错的，所以还是不用实现的是吗…

然后讲义往后看了一点，竟然还有这么多指令，文件分的还是少了点。

新添的头文件是control.h，内部存放了跳转指令的执行辅助函数。

再次RTFSC

感觉慢慢习惯了阅读这么多代码了。

实现更多的指令
add.c

  • add

    x[rd] = x[rs1] + x[rs2]

  • sltiu

    无符号数小于立即数则置位。比较x[rsl]和有符号扩展的immediate，比较时视为无符号数。如果x[rsl]更小，向x[rd]写入1，否则写入0。

    x[rd] = (x[rs1] < unsighed(sext(immediate))

  • bne

    if(rs1 != rs2) pc+=sext(offset)

  • beqz

    伪指令，查阅手册得知功能为：

    If(rs1 == 0) pc += sext(offset)

    可视为 beq rs1, x0, offset

  • beq

    If(rs1 == rs2) pc += sext(offset)
